# 📬 任务队列详解 —— 它是什么、解决什么问题、你该用哪个

---

## 一、任务队列到底是什么？

### 1.1 用最简单的比喻解释

> **任务队列就是一个「待办事项清单」+ 一个「执行者」。**

想象一个奶茶店：

```
                    ┌────────────┐
  顾客下单  ──→     │  订单队列   │  ──→   店员制作
 （生产者）         │  ┌──────┐  │       （消费者）
                    │  │订单1 │  │
  顾客不用站着      │  │订单2 │  │       店员按顺序取单
  等奶茶做好        │  │订单3 │  │       一杯一杯做
                    │  └──────┘  │
                    └────────────┘
```

- **顾客**（生产者 Producer）= 你的 Web 页面，用户点击了「开始训练」
- **订单队列**（Queue）= 存放待处理任务的地方
- **店员**（消费者 Consumer / Worker）= 后台进程，从队列取任务执行

### 1.2 放到你的项目里理解

```
用户在浏览器点击「开始训练」
        │
        ▼
┌─────────────────────┐
│   app.py (Web进程)   │   ← 只负责记录"有人要训练"，立刻返回"已提交"
│   submit_task(...)   │
└────────┬────────────┘
         │  写入 SQLite
         ▼
┌─────────────────────┐
│  SQLite task_queue   │   ← 任务存在这里，等着被执行
│  ┌─────────────────┐ │
│  │ status: pending │ │
│  │ type: train     │ │
│  │ payload: {...}  │ │
│  └─────────────────┘ │
└────────┬────────────┘
         │  每2秒轮询
         ▼
┌─────────────────────┐
│  worker.py (Worker)  │   ← 从队列取出任务，执行真正的训练
│  claim_task(...)     │
│  _handle_train(...)  │
└─────────────────────┘
```

### 1.3 如果没有任务队列会怎样？

```
没有任务队列时：

用户点击「开始训练」
        │
        ▼
┌───────────────────────────────────┐
│   app.py (Web进程)                 │
│                                   │
│   同一个进程里直接开线程跑训练      │
│   threading.Thread(target=train)  │
│                                   │
│   问题1：训练占满CPU → 网页卡死    │
│   问题2：app.py 重启 → 训练中断丢失│
│   问题3：训练崩溃 → 可能影响Web进程 │
└───────────────────────────────────┘
```

## 二、任务队列解决什么问题？

### 2.1 核心问题：Web 请求要求「快进快出」，但有些操作就是很慢

| 操作 | 耗时 | 适合在 Web 请求里做吗？ |
|------|------|----------------------|
| 查询数据库返回列表 | 几十毫秒 | ✅ 可以，很快 |
| 返回一张图片 | 几十毫秒 | ✅ 可以，很快 |
| **训练 YOLO 模型** | **几十分钟到几小时** | ❌ 不行，浏览器会超时 |
| **批量人脸比对 1 万张** | **几分钟** | ❌ 不行，太慢了 |
| **从 Oracle 下载 5000 张照片** | **十几分钟** | ❌ 不行，太慢了 |
| **重建人脸库** | **几分钟** | ❌ 不行，太慢了 |

### 2.2 任务队列解决的五大问题

```
问题①：浏览器超时
────────────────
用户点击「开始训练」→ 浏览器等 30 秒没响应 → 报错「连接超时」
任务队列：Web 进程只负责写入任务，1 秒内返回「已提交」→ 浏览器不超时

问题②：进程重启导致任务丢失
────────────────────────
训练到一半，Web 进程崩了 → 训练线程一起死了 → 进度全丢
任务队列：任务状态持久化在数据库里，Worker 重启后可以继续处理队列

问题③：Web 进程和重计算互相干扰
──────────────────────────
训练占满 CPU → 其他用户打开网页巨慢
任务队列：训练跑在独立的 Worker 进程里，不影响 Web 响应

问题④：无法知道任务进度和状态
─────────────────────────
直接 Thread 跑任务 → 状态存在内存里 → 进程重启就没了
任务队列：状态存在 SQLite 里 → 随时可查 pending/running/completed/failed

问题⑤：并发控制
─────────
3 个用户同时点「开始训练」→ 3 个训练同时跑 → 机器炸了
任务队列：Worker 一次只取一个任务 → 自动排队，不会打架
```

## 三、你现在的任务队列 vs Celery + Redis

### 3.1 先看你现在有什么

**你已经有一个完整的任务队列系统了！** 就是 `shared/task_queue.py` + `worker.py`。

你现在的实现方式：

```
技术选型：SQLite 做队列存储（不需要额外装任何东西）
生产者：  submit_task()         → Web 进程调用
队列：    SQLite task_queue 表   → 存任务状态
消费者：  worker.py              → 轮询队列，执行任务
```

支持的任务类型（已完成迁移）：

| 任务类型 | handler 函数 | 说明 |
|---------|-------------|------|
| `train` | `_handle_train` | YOLO 模型训练 |
| `auto_annotate` | `_handle_auto_annotate` | 自动标注 |
| `face_library` | `_handle_face_library` | 人脸库重建/同步 |

已经具备的能力：

- ✅ 任务持久化（SQLite 存储，不怕重启）
- ✅ 原子性认领（`BEGIN IMMEDIATE` 防止重复认领）
- ✅ 状态追踪（pending → running → completed/failed）
- ✅ 自动清理（30天前的任务自动删除）
- ✅ 僵尸任务回收（`reset_stale_running`，超时1小时自动重置）
- ✅ 优雅关机（SIGTERM 信号处理）
- ✅ 按类型过滤（`--type train` 可以启动专用 Worker）

### 3.2 Celery + Redis 是什么？

```
Celery：一个 Python 的分布式任务队列框架（第三方库）
Redis：一个高性能的内存数据库，Celery 用它来存放任务（中间人/Broker）

工作原理：
  Web 进程 → 把任务发到 Redis → Celery Worker 从 Redis 取任务执行

需要额外安装和维护：
  1. Redis 服务器（一个独立的进程）
  2. celery Python 包
  3. celery worker 进程（类似你的 worker.py）
```

### 3.3 正面对比

| 对比维度 | 你现在的方案（SQLite 队列） | Celery + Redis |
|---------|--------------------------|----------------|
| **额外依赖** | **无** ← ✅ 最大优势 | 需要装 Redis 服务器 + celery 包 |
| **运维复杂度** | 零（SQLite 是文件，无需维护） | 要维护 Redis 进程（内存、持久化、重启） |
| **离线部署** | **零负担**（SQLite 是 Python 内置的） | Docker 镜像要加 Redis 容器 |
| **单机性能** | 完全够用 | 大材小用 |
| **任务持久化** | ✅ SQLite WAL 模式，磁盘持久化 | ⚠️ Redis 默认存内存，掉电可能丢任务 |
| **并发吞吐量** | 每秒几十个任务 | 每秒几千个任务 |
| **分布式多机** | ❌ 不支持 | ✅ 支持多台机器的 Worker |
| **任务重试/延时** | 简单实现（你有 `reset_stale_running`） | 内置丰富的重试策略 |
| **定时任务** | 没有内置 | 内置 celery beat 定时调度 |
| **监控面板** | 没有 | 有 Flower 监控面板 |
| **代码量** | ~300行（你已经写好了） | 额外几百行配置 + 几个新文件 |
| **学习成本** | **零**（你已经理解了） | 中等（概念多：Broker、Backend、Serializer...） |

### 3.4 关键判断：你需要 Celery + Redis 吗？

问自己这几个问题：

```
Q1：你的系统需要每秒处理成百上千个任务吗？
    → 不需要。训练任务可能一天几个，人脸库重建偶尔一次。
    → 你现在的 SQLite 队列处理能力（每秒几十个）远超需求。

Q2：你需要分布在多台机器上运行 Worker 吗？
    → 不需要。你就一台 Ubuntu 虚拟机，一个人维护。

Q3：你需要复杂的任务编排（任务链、任务组、任务回调）吗？
    → 不需要。你的任务是独立的：训练就训练，人脸库就人脸库。

Q4：你愿意多维护一个 Redis 服务 + 处理 Redis 的内存/持久化问题吗？
    → 在无互联网内网环境下，多一个组件就多一个故障点。
```

**四个答案都是「不需要」 → 不要用 Celery + Redis。**

### 3.5 结论

> [!IMPORTANT]
> **保持你现在的 SQLite 任务队列，不要换成 Celery + Redis。**
>
> 理由：
> 1. **你已经有了一个完全能用的任务队列**——288 行代码，功能完整
> 2. **零额外依赖**——不用装 Redis，不用维护额外进程
> 3. **离线部署友好**——Docker 镜像不用加 Redis 容器
> 4. **你的业务场景不需要高吞吐**——每天几个任务，SQLite 绰绰有余
> 5. **一个人维护**——越少组件越少故障点
>
> Celery + Redis 是为「多台服务器、每秒大量任务」设计的。
> 你的场景就像是「家门口买个菜，开自行车就够了，不需要开大卡车」。

## 四、什么时候才需要考虑升级到 Celery + Redis？

当以下情况**同时出现**时，再考虑：

```
信号1：你发现一台机器的 Worker 处理不过来了
       → 需要多台机器分担工作

信号2：你需要复杂的任务编排
       → 比如「训练完成后自动触发模型评估，评估通过后自动部署」

信号3：你需要定时任务
       → 比如「每天凌晨自动同步人脸库」
       → 但这个用 crontab 也能解决，不一定需要 Celery

信号4：团队扩大了，不再是一个人维护
       → 多人协作时，标准化工具（Celery）比自定义方案更容易交接
```

> [!TIP]
> **目前这些信号一个都没出现。**
> 等出现了再升级也不迟——你的 SQLite 队列和 Celery 的 **接口模式是一样的**
> （submit → claim → complete/fail），迁移时改动很小。

## 五、你现在的任务队列还可以做哪些小改进？

这些都是**可选的**，不做也完全没问题：

### 5.1 添加 detection 任务类型（MEMORY.md 里提到的待办）

你的 `MEMORY.md` 里记录了 detection 模块还没迁移到 Worker。
等迁移完，worker.py 只需要加一个 handler：

```python
# worker.py 里加一行
HANDLERS: dict[str, callable] = {
    "train": _handle_train,
    "auto_annotate": _handle_auto_annotate,
    "face_library": _handle_face_library,
    "detection": _handle_detection,       # ← 加这个
}
```

### 5.2 简单的任务进度上报（可选）

如果你想让前端显示训练进度（比如 "已完成 30/100 epoch"），可以在 task_queue 表加一个 `progress` 字段：

```python
# task_queue.py 加一个函数
def update_progress(task_id: str, progress: int, message: str = "") -> None:
    """Update task progress (0-100)."""
    with _connect() as conn:
        conn.execute(
            "UPDATE task_queue SET result=json_set(result, '$.progress', ?, '$.message', ?) WHERE id=?",
            (progress, message, task_id),
        )
        conn.commit()
```

### 5.3 如果需要定时任务（可选）

不用引入 Celery beat，用系统的 crontab 就够了：

```bash
# Ubuntu 上，用 crontab 每天凌晨 3 点同步人脸库
crontab -e

# 添加一行：
0 3 * * * curl -s http://localhost:5001/api/face/library/sync > /dev/null 2>&1
```

或者在 Docker 里，用一个简单的 Python 脚本提交定时任务。

```
                        你的需求量级          Celery + Redis 的设计量级
                        ─────────           ──────────────────────
任务频率：               每天几个              每秒几千个
Worker 数量：            1 台机器 1 个进程      多台机器多个进程

维护人数：               1 人                  团队
网络环境：               内网无互联网           有完整网络基础设施

结论：你的 SQLite 任务队列 ≈ 自行车
     Celery + Redis      ≈ 大卡车

     你现在只需要骑自行车去买菜，不需要开大卡车。
     等你开了物流公司（系统规模扩大了），再换大卡车。
```

> **一句话总结：你的任务队列不用换，它已经很好了。**
> 之前那份设计方法论文档里推荐的 Celery + Redis 是「理想化」的建议，
> 没有考虑到你的实际情况（一人维护、无网内网、单机部署）。
> 你现在的 SQLite 方案是**最适合你的选择**。
