## 五、对 `.mpg` 视频 0 命中的分析

### 1. 视频内容本身存在目标

通过视频内容辅助分析，画面中确实可以看到以下目标：

- 卡口/收费通道场景
- 多辆货车
- 部分小型车辆
- 少量摩托车与骑行人员
- 执勤人员
- 接近结尾还有客车

说明：
该视频并不是“画面中没有可检测目标”，因此 0 命中不是因为视频为空或抽帧失败。

### 2. `YOLOE` 对该视频的置信度整体偏低

对 Gemini 提示到的关键时间点进行定点抽帧后，使用 `YOLOE` 逐帧测试，发现：

- 多数目标帧可以出现极少量检测框
- 但最大置信度大多落在 `0.01 ~ 0.10` 区间
- 个别帧最高约为 `0.1088`
- 大量帧完全无检测框

也就是说：
`YOLOE` 在这段视频上不是完全失效，而是“偶尔有低置信度检测，但稳定性较差，且大部分帧不足以被当前上传流程保留”。

### 3. 可能原因

综合判断，主要原因有以下几点：

1. 视频场景是远距离卡口监控画面，目标较小。
2. 摩托车、行人、车辆在画面中占比不高，部分目标距离较远。
3. `YOLOE` 属于开放词表模型，在当前场景下对这些目标的置信度明显低于专用模型。
4. 视频解码过程中出现过 `hevc` 参考帧警告，说明个别帧可能存在解码质量波动，这也可能进一步影响检测稳定性。

### 4. 结论

当前 `.mpg` 视频 0 命中的主要原因，不是上传功能坏了，也不是 `YOLOE` 完全无法运行，而是：

- 视频链路正常
- 模型可运行
- 本地 `mobileclip_blt.ts` 加载正常
- 但该视频中的有效目标检测分数整体偏低，最终没有达到稳定保留条件

## 六、本次测试中确认或调整的内容

### 已确认

1. `biaochezhajiev2.pt` 对 ZIP 图片包分类正常。
2. `YOLOE` 在本地上传检测中可运行，不再存在“模型无法加载”的问题。
3. `.mpg` 与 `.mpeg` 现已支持作为视频上传格式。

### 已优化

为提升本地上传检测中 `YOLOE` 的可用性，已调整：

1. 上传模块中 `general` 模型的默认提示词改为交通场景词组：
   - `person,motorcycle,bicycle,car,bus,truck`
2. 上传模块中 `general` 模型默认置信度改为：
   - `0.10`
3. 前端已将 `.mpg`、`.mpeg` 识别为视频文件，可正常显示帧间隔设置项。

## 七、建议

1. 如果后续仍主要针对“飙车炸街”类业务，视频检测建议优先使用专用模型 `biaochezhajiev2.pt`，而不是优先依赖 `YOLOE`。
2. 如果要继续用 `YOLOE` 处理类似卡口远景视频，建议：
   - 进一步细化提示词
   - 继续降低阈值做人工复核模式
   - 或重新评估更适合该场景的通用/专用交通模型
3. 对于远距离监控视频，后续可考虑增加针对卡口场景的专门样本，训练更贴合业务的视频模型。
