# 人脸识别身份集成开发清单

## 1. 目标

- 在当前项目“本地上传检测”和“Oracle / 数据库检测”流程中加入身份识别能力。
- 保持现有目标检测流程不变：用户仍然先上传 ZIP 图片包或视频，完成目标检测后得到筛选结果。
- 在检测完成后，允许用户在页面中选择一张或多张结果图片，并发起人脸身份识别。
- 技术实现参考 `service/0312face_recognition_pipeline.py`。
- 人脸特征库继续使用你之前设计的内网 SQL 作为数据源。

## 2. 当前项目基础情况

- 当前本地上传功能已经支持 ZIP 图片包和视频文件。
- 检测入口在 `routes/upload_routes.py`。
- 后台任务逻辑在 `service/upload_job_service.py`。
- 本地上传页面在 `templates/index.html`。
- 独立的人脸识别参考脚本在 `service/0312face_recognition_pipeline.py`。

## 3. 目标用户流程

1. 用户在“数据库检测 / Oracle 检测”页或“本地上传检测”页完成目标检测。
2. 系统生成筛选后的结果图集合。
3. 页面展示筛选出的结果图片，而不是只有下载链接。
4. 用户可以勾选一张或多张结果图。
5. 用户点击“识别身份”或“批量识别身份”按钮。
6. 后端对每张选中图片做人脸检测，提取一张或多张人脸特征。
7. 后端用人脸特征库做比对，返回每张图片对应的身份候选结果。
8. 页面展示逐图身份结果、相似度分数、人脸质量提示以及“匹配 / 无匹配 / 无人脸 / 低质量”等状态。

## 4. 范围说明

### 本期纳入范围

- 针对“本地上传检测”结果图做身份识别。
- 针对“数据库检测 / Oracle 检测”结果图做身份识别。
- 从内网 SQL 同步人脸底库，并在本地生成缓存。
- 本地保存底库照片和人脸特征。
- 在网页中完成结果图片单选 / 多选识别和结果展示。
- 提供基础的人脸库同步和重建能力。

### 本期暂不纳入

- 完整权限系统。
- 在原始上传文件上直接做人脸检索。
- 对所有筛选结果自动批量做人脸身份识别。
- Figma 自动出代码。

## 6. 后端架构方案

### 6.1 新增服务模块

- `service/face_identity_service.py`
  - 加载人脸模型
  - 执行人脸检测和特征提取
  - 执行 1:1 / 1:N 比对
- `service/face_library_service.py`
  - 从内网 SQL 同步人员信息
  - 解码并保存底库照片
  - 生成人脸特征并管理本地缓存
  - 维护底库状态信息
- `service/face_result_service.py`
  - 管理 Oracle 检测结果图、本地上传检测结果图与后续身份识别结果的绑定关系

### 6.2 复用 `0312face_recognition_pipeline.py` 的内容

- `FaceDetector`
- `FaceRecognizer`
- `extract_probe_embeddings()`
- `search_face()`
- `verify_face()`
- SCRFD 的 ONNX 动态输出补丁逻辑
- 人脸缓存目录结构：
  - `face_data/photos`
  - `face_data/features`
  - `face_data/person_db.pkl`

### 6.3 重构要求

- 不要继续把所有逻辑放在一个脚本里。
- 把人脸模型加载、特征提取、检索比对拆成可复用服务。
- 把脚本式调用改造成适合 Flask 接口调用的服务层。

## 7. 人脸特征库方案

### 7.1 数据来源

- 人脸底库来源继续使用内网 SQL。
- 复用之前已经设计好的 SQL。
- SQL、账号、密码、连接信息放进配置，不直接硬编码在服务代码里。

### 7.2 本地缓存

- 本地保存底库照片。
- 每个人保存一个 embedding 特征文件。
- 保存一个序列化后的人员索引文件。
- 增加一份轻量状态文件，记录：
  - 最近同步时间
  - SQL 拉取记录数
  - 有效照片数
  - 有效特征数
  - 使用的人脸模型版本

### 7.3 刷新模式

- 全量同步 SQL 并重建底库。
- 不重新查 SQL、只重建特征。
- 增量同步可放到后续版本。

## 8. 接口设计

### 8.1 新接口

- `POST /face-library/sync`
  - 从内网 SQL 同步人脸底库
- `POST /face-library/rebuild`
  - 基于本地照片重建特征
- `GET /face-library/status`
  - 查看最近同步状态、数量和模型就绪情况
- `POST /face/identify`
  - 输入：一张或多张检测结果图
  - 输出：每张图片中的人脸信息和身份候选结果
- `POST /face/identify/batch`
  - 可选独立批量接口，用于大批量多选识别
- `POST /face/verify`
  - 可选后续接口，用于指定身份的 1:1 核验

### 8.2 输入方式

- 不建议让用户重复上传同一张图片用于身份识别。
- 应直接复用检测完成后保存到本地的结果图片。
- 同时支持 Oracle 检测结果图和本地上传结果图。
- 页面通过结果图片 ID、路径标识或任务结果引用来触发识别。

### 8.3 输出结构

- `status`: `matched | no_match | no_face | low_quality | error`
- `face_count`
- `faces[]`
  - `bbox`
  - `blur_score`
  - `det_score`
  - `used_align`
  - `top_matches[]`
    - `name`
    - `id_number`
    - `score`
    - `photo_path`

## 10. 结果图和存储管理

- 结果图不能只打包成 ZIP，还需要保留可浏览的本地目录结构。
- Oracle 检测结果和本地上传结果应共用统一的结果 manifest 结构。
- 每个上传任务需要保存一份 manifest，至少包括：
  - 任务 ID
  - 源类型
  - 图片文件名
  - 本地路径
  - 缩略图路径（如有）
  - 后续身份识别结果
- 这份 manifest 建议落到 SQLite 或本地 JSON 中统一管理。

## 11. 模型与运行要求

### 11.1 需要的人脸模型

- SCRFD 人脸检测 ONNX 模型
- ArcFace 人脸特征 ONNX 模型

### 11.2 配置项建议

- `FACE_MODEL_DET`
- `FACE_MODEL_REC`
- `FACE_DATA_DIR`
- `FACE_SIMILARITY_THR`
- `FACE_BLUR_THRESH`
- `FACE_SQL_ENABLED`
- `FACE_SQL_HOST`
- `FACE_SQL_PORT`
- `FACE_SQL_DB`
- `FACE_SQL_USER`
- `FACE_SQL_PASSWORD`
- `FACE_SQL_QUERY_PATH`

### 11.3 环境说明

- 内网 SQL 在当前环境里无法直接联通，所以同步接口必须允许失败并给出明确提示。
- 即使 SQL 暂时不可用，也应允许系统继续使用本地缓存人脸库完成识别。

## 12. 安全与合规

- SQL 密码不要写进 Git 跟踪文件。
- 页面上是否展示完整证件号，需要按实际需求控制。
- 记录操作审计日志：
  - 人脸库同步
  - 重建特征
  - 身份识别请求
- 对结果图和识别结果设置清理策略。

## 14. 验收标准

- 用户仍然可以正常完成现有目标检测流程，不能有回归。
- 用户不下载结果文件，也能在页面中直接浏览筛选结果图。
- 用户可以选择一张或多张结果图并发起身份识别。
- 系统能返回候选身份和相似度分数。
- 系统能明确区分“无人脸 / 无匹配 / 低质量 / 底库未就绪”等状态。
- 内网 SQL 可用于构建人脸库，本地缓存可支持后续离线识别。
- 当预处理或模型变化后，系统支持重建特征。
- Oracle 检测链路和本地上传检测链路都具备身份识别能力。
